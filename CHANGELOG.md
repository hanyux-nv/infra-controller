# Changelog

All notable changes to this project will be documented in this file.

The format is based on [Keep a Changelog](https://keepachangelog.com/en/1.1.0/),
and this project adheres to [Semantic Versioning](https://semver.org/spec/v2.0.0.html).

## [v2.0.0]

### Added

- Added a bypass-state-controller flag to the component management API, letting clients dispatch power and firmware control commands directly to the underlying component backend (RMS, NSM, PSM, etc.) instead of routing through the state controller. This gives operators a faster, more direct path for component control when the standard managed workflow isn't needed. (#1803)
- Added an artifact pre-caching pipeline and HTTP cache server to RVS (RVS resolves direct URIs and JSONPath-based sotpath expressions, downloads artifacts concurrently to a local cache, and serves them to nodes during validation), reducing repeated external downloads during validation cycles. Downloads are concurrency-limited, timeout-bounded, and verified against SHA-256 checksums, and previously cached files are skipped on later cycles; this also reflects that SOT firmware data is now push-only into RMS rather than pulled from NICo over gRPC. (#1813)
- Added full API, database, admin CLI, and admin web UI support for NVLink domain health records, including new list/insert/remove RPCs, a dedicated nvlink_domain_health_reports table, and RBAC permissions. Operators can now manage NVLink domain health reports via `carbide-admin-cli nvl-domain health-report show|remove|print-empty-template` or through paginated, searchable admin web UI pages. See https://nvidia.github.io/infra-controller/ for the schema/updated schema. (#1854)
- Added an NVOS gNMI telemetry collector (disabled by default) that streams component, interface, and system-event data from switches over long-lived, auto-reconnecting gRPC streams, with output to Prometheus or OTLP sinks. (#1975)
- Added a deployment guide for installing and configuring DPF (DPU provisioning backend) for use with NICo, covering prerequisites, chart parameter overrides, and required post-install cluster state. The guide is now linked under Getting Started > Installation Options in the documentation. (#1999)
- The nico-prereqs Helm chart now exposes a postgresql.storageClass value so operators can choose the StorageClass used by the Zalando-managed NICo Postgres cluster, and setup.sh gains a NICO_MANAGE_DEFAULT_STORAGE_CLASS toggle to control whether local-path is annotated as the cluster's default StorageClass. Existing deployments are unaffected since the default storage class behavior is unchanged unless operators opt in. (#2034)
- NICo now records and exposes the last-seen forge-scout version for each machine as a new last_seen_scout_version field, populated when forge-scout reports its version during machine discovery and updated on each subsequent re-discovery. Existing machines will show null for this field until their next discovery. See https://nvidia.github.io/infra-controller/ for the schema/updated schema. (#2037)
- Added an admin API endpoint to force-delete racks, giving administrators a way to remove rack records that could not otherwise be cleaned up through the normal deletion flow. See https://nvidia.github.io/infra-controller/ for the schema/updated schema. (#2043)
- Added an optional docker_image_pull_secret field to DpfConfig that, when set, is applied as the imagePullSecrets entry for all mandatory DPF services except dts and doca_hbn (which retain their own configured secrets). This lets operators supply a custom image pull secret for mandatory services without having to configure each one individually, overriding any secret previously set in the services section. (#2074)
- Added site master encryption key (KEK) rotation support for SPIFFE machine identity: operators can update the configured current encryption key ID and re-wrap existing tenant identity secrets via a new admin gRPC, Forge.ReencryptTenantIdentitySecrets, which supports dry-run validation and reports the target key ID used. The per-row encryption_key_id database column was dropped in favor of an envelope-embedded key ID, and several queries were hardened to use explicit column lists instead of SELECT/RETURNING * to avoid Postgres plan-cache errors after schema migrations. (#2136)
- Added a second compute-tray component-manager implementation that drives compute trays through Core's Component Manager dispatch path (ComponentPowerControl, UpdateComponentFirmware, etc.), the same mechanism already used by nvswitch and powershelf. The new path is toggled via an environment variable and defaults to the existing behavior, so no action is required unless operators opt in. (#2142)
- Added a Day 2 Operations guide covering monitoring and health, documenting how operators can observe system health after initial deployment. (#2146)
- Added a Day 2 Operations guide covering Firmware Updates, published in the operations documentation section to help operators plan and execute firmware update procedures. (#2147)
- Added a Day 2 Operations guide covering Tenant Lifecycle Cleanup, published in the operations documentation section to help operators understand the cleanup workflow. (#2148)
- Added power control operations (on, off, reset) to the switch-controller, allowing switches to be power-cycled or toggled directly rather than requiring manual intervention. (#2157)
- Added a transceiver-exporter sidecar (mirroring the existing node-exporter sidecar) that reads module EEPROM/DOM data via ethtool netlink over the host network namespace and exposes it on a loopback-only port. Since no upstream container image exists, the sidecar is built and published by CI as a new arm64 image alongside otelcol-contrib, giving operators visibility into transceiver health metrics. (#2160)
- Added a shared rule validation framework providing generic rule envelopes, selectors, scopes, and bindings, along with a kind validator registry. Kind-specific validation logic is isolated behind a narrow KindValidator interface, laying the groundwork for future rule resolvers and executors. (#2162)
- Added a nico-ntp Helm subchart that deploys a 3-replica chrony NTP StatefulSet with per-pod LoadBalancer VIPs, completing end-to-end NTP service delivery for DPUs whose VIPs were already advertised by the DHCP hook but previously had nothing listening. Also fixed a template scoping bug in nico-dns's external-service range loop and expanded the Helm prerequisites documentation. (#2167)
- Health metrics can now be tagged with a machine_id, making it possible to associate specific health metric readings with the machine that produced them. (#2168)
- Added a consolidated Configurability guide documenting NICo's four configuration layers (Helm values, nico-api siteConfig TOML, environment variables/secrets, and external systems) with pointers to the canonical in-repo references for each. This is a documentation-only addition to help operators find and understand configuration options. (#2182)
- Added support for configuring multicast group limits on NVLink partitions. NICo now deletes domain default partitions after discovery and moves GPUs into per-tray default partitions (TDPs), which are tracked only on the NMX-C and used to manage GPU placement as instances and tenant NVLink configurations are created, updated, or removed; all partitions are created with a multicast group limit attribute of 18. (#2191)
- Introduces a new, separately packaged and containerized MCP (Model Context Protocol) server that exposes read-only NICo REST API operations, launched automatically alongside other services when running `make kind-reset`. Each connection parameter can be supplied explicitly, or the server can be preconfigured to talk to a single known REST API endpoint, making it easier to integrate NICo data into MCP-compatible tools. This is phase 1 (read-only commands only); write operations are not yet supported. (#2194)
- Preingestion now performs a mandatory one-time BMC reset immediately after the initial state, before proceeding with existing time-sync and firmware checks. This addresses cases where GB200 host BMCs intermittently drop the second DPU from their Redfish PCIe inventory, ensuring both DPUs are reliably detected when a managed host is created. (#2197)
- Wired power shelf and switch controllers through the component-manager, extending controller-based management to these component types. (#2198)
- Introduced a lifecycle for VPC prefixes: deleting a VPC prefix now performs a soft-delete first, becoming a hard-delete only once it is no longer referenced by any network segments. Attempting to delete a VPC prefix that is still referenced by a network segment now returns a clear error instead of an ambiguous failure, and VPCs cannot be deleted while any of their VPC prefixes are still pending clean-up. See https://nvidia.github.io/infra-controller/ for the schema/updated schema. (#2199)
- Added a --zero-dpu flag to admin-cli that allows instance allocation onto hosts without DPUs. When set, machine selection no longer rejects DPU-less hosts, interface matching no longer requires Mellanox PCI properties, and the InstanceAllocationRequest is sent with auto-interface selection instead of explicit interfaces. (#2206)
- Added initial core support for the Astra platform alongside the existing SVPC platform, laying groundwork for future Astra-specific functionality while keeping SVPC support complete. (#2216)
- Extended the DpuInfo API response with additional firmware, OS, and network representor state details, including new structs for OS and representor state that distinguish "admin down" from "carrier down" conditions. These structs are designed to be extended further as more state details become available. See https://nvidia.github.io/infra-controller/ for the schema/updated schema. (#2234)
- Added opt-in legacy carbide/forge hostname aliases for nico-api services via ExternalName Kubernetes Services, so binaries that still hardcode pre-rename service URLs continue to resolve correctly without needing a rebuild. Both the legacy and current names now work simultaneously, pointing at the same pods. (#2249)
- NICo now stores the complete identity of a host's boot interface -- MAC address plus its Redfish interface ID, in a new MachineBootInterface structure, instead of relying on the MAC address alone. Boot device management calls still try the MAC first but now fall back to the stored interface ID, making boot-order management resilient to BIOSes that drop a DPU's MAC from Redfish after a DPU-to-NIC mode switch; the admin UI also gains a new "Restore Boot Interface" action to recover machines that were previously left in an unrecoverable state. See https://nvidia.github.io/infra-controller/ for the schema/updated schema. (#2261)
- Site operators can now enable OTLP tracing by default and lock it from being toggled at runtime via the `nico-api` config TOML, and the OTLP endpoint can now also be set there, with the existing environment variable still available as an override. (#2263)
- Added a `ComponentStatus` model that provides a unified, per-component readiness view (phase, reason, and blocked operations) derived from each component type's underlying state machine (compute, switch, power shelf). This status is recomputed each inventory sync cycle, persisted, and now surfaced on every `Component` response from the gRPC API, giving consumers a consistent way to check whether power or firmware control operations are currently allowed on a component. See https://nvidia.github.io/infra-controller/ for the schema/updated schema. (#2264)
- Added a user manual for the DPU Extension Service, covering what these services are, when to use them, and how to manage their lifecycle. (#2269)
- Flow now logs every gRPC call on both its server and client paths via new unary interceptors, emitting one structured log line per RPC with service, method, status code, duration, and peer/target (no sensitive payloads such as BMC credentials are logged). Additionally, force-power-on Stage 4 timeouts were raised to 20 minutes per stage and 5 minutes per action across all twelve entries, up from 4/3 minutes, to accommodate slower PowerShelf and NVSwitch power-on sequences. (#2276)
- Added user documentation for Machine Identity (SPIFFE), explaining the settings and configuration needed to enable machine identity features. (#2282)
- Added a configurable `suppress_unchanged_interval` setting (default 5 minutes) on the health report sink that suppresses re-sending success-only health reports whose content hasn't changed within the interval, reducing unnecessary API and network load. Reports containing any alert always bypass suppression and are forwarded immediately; setting the interval to null disables suppression entirely. (#2284)
- Managed host interfaces now carry a full `MachineBootInterface` (MAC address paired with its Redfish-reported interface ID) rather than just a MAC address. This lets the system reconcile boot device configuration by interface ID even in cases where a DPU's MAC address disappears from Redfish after switching to NIC mode. (#2285)
- Added Task Rule REST APIs for managing Flow Operation Rules, including five new site-scoped endpoints under `/v2/org/{org}/nico/task/rule` for creating, listing, retrieving, updating, and deleting rules. Also added an optional `ruleId` field to eight rack/tray power, firmware, and bring-up request schemas (single and batch) to let callers pin a specific rule, and a `ruleId` field on the `RackTask` response schema so callers can see which rule was applied to a task. Existing callers that omit `ruleId` see no behavior change, since Flow continues to fall back to its default rule resolution. See https://nvidia.github.io/infra-controller/ for the schema/updated schema. (#2286)
- The Task API now surfaces Flow's structured execution report on rack task endpoints. The single-task `GET /rack/task/{id}` and `POST /rack/task/{id}/cancel` endpoints always include the report, while the list endpoints `/rack/{id}/task` and `/tray/{id}/task` gain a new opt-in `withReport=true` query parameter, so callers who don't need the (potentially multi-KB) report can avoid the added payload size. See https://nvidia.github.io/infra-controller/ for the schema/updated schema. (#2288)
- The REST API now supports selective hot-loading of individual configuration attributes, with the site.phoneHomeUrl setting as the first field wired up to reload automatically when the config file changes. Attributes requiring connection suspension (such as database and Temporal settings) are intentionally excluded from hot-reloading, and the previous valid phoneHomeUrl is preserved if a reload encounters an empty or unparseable value. (#2289)
- The admin web UI now displays a warning banner on the Explored Endpoints pages and the root admin page whenever a site-wide default credential (BMC root, host UEFI, or DPU UEFI password) is unset. Previously this condition only produced a single easily-missed ERROR log line, making it hard for operators to notice why endpoint exploration was failing; the banner now lists exactly which credentials are missing and points to the relevant nico-admin-cli commands to set them. (#2292)
- The admin web UI now supports adding and removing a SKU assignment directly on a machine's detail page, removing the need to use the CLI or API for this operation. (#2295)
- Site exploration now records the Redfish interface ID alongside the MAC address for every host NIC, not just DPU interfaces, extending this to SuperNICs and integrated NICs as well. This lays the groundwork for any host network interface to be usable as a first-class boot interface. (#2311)
- Added a new `SetPrimaryInterface` RPC and matching `managed-host set-primary-interface <host> <interface-id>` CLI command, allowing any host network interface (not only a DPU) to be designated as the primary boot interface, identified by its `MachineInterfaceId`. This enables promoting a boot interface even on "zero-DPU" hosts that have no DPU available. (#2314)
- LiteOn power shelves and NVIDIA switches are now registered as their proper expected inventory types (expected switches and expected power shelves) rather than as generic expected machines, and switch NVOS MAC/serial data is now preserved across persistence. Mock BMC handling was also adjusted to support BMC-only hardware of these types. (#2320)
- Added support for ingesting delta powershelf data, allowing incremental power-shelf updates to be processed rather than requiring full re-ingestion. (#2322)
- Added the `machine hardware-info show` admin-cli command, allowing operators to view detailed hardware information for a given machine from the command line. (#2329)
- Added SRE-oriented troubleshooting playbooks covering stuck-object diagnosis and production operations, split into focused pages for state machine debugging, diagnostic tools, DPU provisioning, host ingestion, health alerts, connectivity, instance/fabric issues, and site-controller health. These new pages are wired into the Operations (Day 2) documentation navigation to help operators resolve issues faster. (#2334)
- Added documentation for the NICo Debug WebUI, including a note on its basic authentication requirements. (#2346)
- Structured (logfmt) log lines now include a component field identifying the emitting binary or, for in-process subsystems of the API server (e.g. site-explorer, machine_state_controller, switch_controller), the specific subsystem. This allows operators to filter centralized logs by component instead of relying on free-text message matching. (#2354)
- Added a configurable `host_naming_strategy` option that controls how machine hostnames are generated, supporting IP address (default, matching prior behavior), serial number, MAC address, or a whimsical adjective-noun scheme. The strategy is consulted whenever a hostname is assigned or updated, such as on initial DHCP or address changes. (#2361)
- carbide-api-config now supports defining VPCs directly in the configuration TOML and attaching a vpc_name property to network definitions, making it easier to configure a working flat, zero-DPU network topology for sites with a single flat subnet of DPU-less hosts entirely from the config file. - Hardware health monitoring now collects non-sensor metrics (scalar and other values exposed via the Redfish Metrics endpoint) in addition to sensor-backed data. The new metrics collector is disabled by default for this release; OEM metrics support is planned as a follow-up. (#2433)
- Introduced a lightweight, dependency-free nico-test-support crate providing a table-driven testing pattern, with example usage added across several crates. This is an internal testing utility with no impact on runtime behavior. (#2437)
- Flow can now mirror Core's expected inventory (racks, machines, switches, and power shelves) into its own database tables on every inventory cycle, ahead of drift detection, gated behind the FLOW_EXPECTED_INVENTORY_SYNC_ENABLED environment variable which defaults to off. When enabled, racks are matched by Core's rack_id and components by manufacturer/serial number, soft-deleted rows resurrect on re-report to keep UUIDs stable, and per-type sync is fault-tolerant so a transient RPC error or empty response from Core does not wipe existing Flow data. (#2440)

Soft-deleted prefixes are now immediately reported with a terminating state rather than appearing unchanged. See https://nvidia.github.io/infra-controller/ for the schema/updated schema. (#2447)
- A new retained_boot_interfaces table preserves a machine's boot target (the MAC address and Ethernet interface ID pairing known as MachineBootInterface) across machine or interface deletion, so operations like force-delete --delete-interfaces no longer strand a re-ingested machine without a boot target. Predicted machine interfaces also gain a boot_interface_id column so hosts explored before their first DHCP request can be matched to a prepared boot interface as soon as one becomes available. (#2448)
- DPF DPU configuration now supports specifying an https_proxy and an optional no_proxy value via the TOML configuration file, allowing DPUs to reach services through an HTTPS proxy. DPUFlavors are now identified using a SHA-256 digest of their spec so that runtime changes to a DPUFlavor can be tracked and accumulated correctly. (#2459)
- The PATCH /tray/{id}/firmware endpoint (and its batch equivalent) now supports an explicit opt-in "dpu" target that triggers DPU reprovisioning on a host's associated DPU machines as part of the firmware update, running a multi-step saga that prevents allocations, triggers reprovisioning, reboots the host, and waits for reprovisioning to complete before clearing the health override. Mixed requests run compute-tray targets first and the DPU target last, and DPU is never included implicitly by an empty/default target list; the per-request firmware version is ignored for this target since Core uses the site-configured version. See https://nvidia.github.io/infra-controller/ for the schema/updated schema. (#2467)

This supports zero-DPU environments where hosts must DHCP into a pre-seeded network segment before any VPC exists, and lets operators retroactively attach that segment to a VPC once instance allocation begins. (#2470)

(#2481)
- Flow now tracks a leak_status field (UNKNOWN, DETECTED, NOT_DETECTED) on components, populated each cycle by the leak-detection loop based on tray-leak-detection health alerts. This gives operators and downstream consumers visibility into coolant leak state per component. (#2491)

(#2528)

Additionally, the underlying ComponentStatus type/proto message was renamed to ComponentOperationStatus for clarity (no database or wire-format migration needed). See https://nvidia.github.io/infra-controller/ for the schema/updated schema. (#2534)
- Added a REST API endpoint to retrieve the DPU machines attached to a host Machine, exposing DPU metadata over the public REST surface instead of requiring the internal gRPC or admin-cli interfaces. See https://nvidia.github.io/infra-controller/ for the schema/updated schema. (#2539)

This helps API consumers understand how to authenticate and authorize requests against NICo's REST endpoints. (#2549)

See https://nvidia.github.io/infra-controller/ for the schema/updated schema. (#2586)

(#2590)

Operators can now diagnose Site Explorer failures directly from the admin UI instead of inspecting API pod logs. (#2591)
- Adds a new site-level `ntp_servers` configuration option that is pushed to BMCs via Redfish during preingestion, plumbed into the `DiscoverDhcp` RPC's `DhcpRecord` response for DHCP Option 42, and added to `ManagedHostNetworkConfigResponse` for use by the DPU agent's DHCP server. Two new read APIs, `ListMachineValidationRunItems` and `GetMachineValidationAttempt`, let consumers inspect validation execution progress. - Adds an optional new secrets storage backend that lets NICo store credentials in Postgres instead of (or alongside) Vault, using envelope encryption: each credential gets its own data encryption key, wrapped by a key encryption key that lives outside the database in NICo or a configured KMS provider. This backend is fully feature-flagged and disabled by default, implements the same interface existing credential consumers already use, and preserves familiar read semantics (newest write wins, empty password reads as no credential), enabling a future migration path away from Vault without requiring any downstream code changes. These new wire, config, and schema fields are inert for existing DHCPv4 behavior; DHCPv6 request handling and Kea/DPU DHCPv6 serving are not yet enabled. See https://nvidia.github.io/infra-controller/ for the schema/updated schema. (#2672)

The IPv4 DHCP option-6 resolver path is unchanged, and the new IPv6 resolver set is currently inert until a DHCPv6 or Router Advertisement delivery mechanism is added to consume it. (#2686)

This is a test-infrastructure addition that enables end-to-end testing of DPU-to-NIC mode transitions; it has no effect on production behavior. (#2689)

This is groundwork for full PTR query support landing in follow-up PRs and is not yet wired into the live DNS resolution path. (#2691)

This is backend groundwork for reverse DNS (PTR) support and is not yet exposed through the public DNS resolution path. (#2711)
- The DNS lookup handler (`LookupRecord` RPC) now answers PTR (reverse DNS) queries by parsing the arpa-form query name into an IP address and resolving it to the owning interface's FQDN, echoing the queried name back in the reply as PowerDNS's remote backend expects. Queries for names that don't parse or that resolve to no known interface now return an empty answer instead of an error, completing the resolver side of reverse DNS support. See https://nvidia.github.io/infra-controller/ for the schema/updated schema. (#2719)
- carbide-dns now accepts and serves PTR (reverse DNS) queries in addition to A and AAAA; previously PTR queries were rejected with NotImp. With this change, a PTR query is forwarded to the API's `LookupRecord` RPC and returned to the client as a proper PTR DNS record, completing end-to-end reverse DNS resolution. (#2720)
- Added a typed `network_segment_type` field on `ExpectedHostNic`, letting operators declare a host NIC's network segment directly instead of relying on the legacy free-form `nic_type` string, which is now considered deprecated but remains supported as a fallback. This is an additive, non-breaking change to the Forge proto/RPC and REST-adjacent config model with no data migration required. See https://nvidia.github.io/infra-controller/ for the schema/updated schema. (#2747)
- Adds a new metric, `carbide_site_explorer_dpu_migration_signals_count`, that instruments each step of the site-explorer DPU NIC-mode migration flow (mismatch detection, mode-set issuance, power-cycle request, and zero-DPU host registration), giving operators visibility into this process that was previously only observable via logs. (#2748)
- Adds dual-stack IPv6 allocation support for FNN (Fully Named Network) segments, routing dynamic IPv6 DHCP allocation through the existing u128 allocator under exclusive segment locking, while IPv4 continues to use its fast SQL path. Also adds support for static external IPv6 address assignments by seeding the reserved static-assignments segment with an IPv6 placeholder prefix, and removes the hard requirement for an IPv4 prefix before SVI refresh on dual-stack or IPv6-only segments. (#2751)
- Adds support for the Lenovo-AMI GB300s server platform and bumps the bundled libredfish dependency to v0.44.12. (#2753)
